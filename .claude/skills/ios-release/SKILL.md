# iOS Release build + on-device install

This repo is an Expo / React Native app (`apps/mobile`, Continuous Native Generation — the
`ios/` dir is **gitignored and regenerated** by `expo prebuild`). A "release build" here is
an `xcodebuild -configuration Release` producing a Hermes-bytecode, code-signed `.app`
installed on a **physical iPhone** via `devicectl`. Use `bun`/`bunx`, never `npm`/`npx`.

## ⚠️ The core gotcha: `expo run:ios` cannot sign a fresh device build

`bunx expo run:ios --device … --configuration Release` **does not work** for a device that
isn't already provisioned: it signs but passes **neither** `-allowProvisioningUpdates`
**nor** `-allowProvisioningDeviceRegistration`, so it dies with either
`No profiles for 'com.fastvibes.huismus' were found` or
`Device … isn't registered in your developer account`. **Drive `xcodebuild` directly**
(step 3) and install/launch with `devicectl`. Don't waste a build cycle on `expo run:ios`.

## Human-only gates — none of these can be done headless

Do these (or ask the user to) **before** / **during** the build; each has bitten a real run:

1. **Apple ID signed into Xcode** — Xcode → Settings → Accounts → "+" → Apple ID. Without a
   registered team, signing has nothing to use. Verify:
   `security find-identity -v -p codesigning` (want an "Apple Development" identity) and the
   plist team key above.
2. **Developer Mode ON** on the iPhone — Settings → Privacy & Security → **Developer Mode** →
   toggle → reboot → confirm. The menu entry only appears **after** a dev build has been
   attempted once, so the first `xcodebuild` will fail with
   `Timed out waiting for all destinations … Developer Mode disabled` — that's expected;
   flip it and re-run.
3. **codesign keychain prompt** — the first `codesign` from a non-GUI shell pops
   *"codesign wants to sign using key … in your keychain"*. The user must click
   **Always Allow** (clicking Deny → `errSecInternalComponent` at the "Embed Pods Frameworks"
   phase, `BUILD FAILED`). Keychain must be unlocked.
4. **iPhone unlocked** — required for both **install** and **launch**, else `devicectl`
   returns `CoreDeviceError 10002` / `FBSOpenApplicationErrorDomain error 7` ("device was
   not, or could not be, unlocked").
5. **Trust the developer** — first open on device may show "Untrusted Developer": Settings →
   General → VPN & Device Management → "Jeroen Esseveld" → **Trust**.

## How env works — baked at bundle time from `.env`

`EXPO_PUBLIC_*` vars are inlined into the JS bundle when the Xcode "Bundle React Native code
and images" phase runs; `@expo/env` auto-loads `apps/mobile/.env`. There are **no mocks** —
the app fetches everything from `EXPO_PUBLIC_API_URL`. Unless told otherwise, **build against
staging**; the committed `.env` already reads:

```
EXPO_PUBLIC_API_URL=https://api-staging.realty-ai.nl
EXPO_PUBLIC_AUTH_ENABLED=true
EXPO_PUBLIC_PLAUSIBLE_ENABLED=true          # analytics on in real device builds
EXPO_PUBLIC_GOOGLE_*_CLIENT_ID=…            # Continue-with-Google
```

The prebuild/build log prints `env: load .env` + `env: export EXPO_PUBLIC_…` — confirm it's
there. (Note: CLAUDE.md and the android skill say `.env.local`; on iOS the real file is
`.env`.)

## 1. Confirm the device is reachable

```bash
xcrun devicectl list devices            # want State = "connected" (or "available (paired)")
xcrun xctrace list devices              # get the HARDWARE UDID (00008xxx-… form) for -destination id=
```

`connected (no DDI)` is fine — the Developer Disk Image auto-mounts during the build. If the
device is `unavailable`, it's likely locked or just rebooted; unlock and re-check. Use the
`00008xxx-…` UDID from `xctrace`, **not** the devicectl "Identifier" UUID.

## 2. Regenerate the native project + inject signing

```bash
cd apps/mobile
bunx expo prebuild --platform ios --clean      # regenerates ios/ (fixes stale bundle id, runs pod install; a few min)

# Inject team + automatic signing into the APP TARGET's two build configs.
# (ios/ is gitignored & regenerated, so re-do this after every prebuild.
#  Use perl — the pbxproj uses tab indentation the Edit tool mishandles.)
perl -i -pe 's/^(\s*)PRODUCT_NAME = Huismus;\n/$1PRODUCT_NAME = Huismus;\n${1}DEVELOPMENT_TEAM = 5W85L569QN;\n${1}CODE_SIGN_STYLE = Automatic;\n/' \
  ios/Huismus.xcodeproj/project.pbxproj
grep -nE "DEVELOPMENT_TEAM|CODE_SIGN_STYLE" ios/Huismus.xcodeproj/project.pbxproj   # expect 2 of each
```

## 3. Build (both provisioning flags), install, launch

Run backgrounded — a cold Release build compiles Hermes + all pods and takes several
minutes. Both flags are required: `-allowProvisioningUpdates` creates the cert/profile,
`-allowProvisioningDeviceRegistration` registers the device on the portal.

```bash
DEVICE=<UDID-from-xctrace>            # e.g. 00008150-001D0C400EBB401C ("Dynatron")
IOS=apps/mobile/ios

xcodebuild \
  -workspace "$IOS/Huismus.xcworkspace" \
  -scheme Huismus \
  -configuration Release \
  -destination "id=$DEVICE" \
  -allowProvisioningUpdates \
  -allowProvisioningDeviceRegistration \
  -derivedDataPath "$IOS/build" \
  COMPILER_INDEX_STORE_ENABLE=NO \
  build

APP="$IOS/build/Build/Products/Release-iphoneos/Huismus.app"
xcrun devicectl device install app --device "$DEVICE" "$APP"
xcrun devicectl device process launch --device "$DEVICE" com.fastvibes.huismus
```

Watch the log (host `sleep` is blocked — use a bounded `until grep -qE …; do sleep 4; done`
loop, not raw polling). Key milestones in order: `env: load .env` → `Auto signing … team(s):
5W85L569QN` (if via expo) / `GatherProvisioningInputs` → many `CompileC`/`SwiftCompile` →
`CodeSign … Huismus.app` → `BUILD SUCCEEDED` → `installationURL: …Huismus.app/` →
`Launched application with com.fastvibes.huismus`.

Common failure → cause (all recoverable, re-run after fixing):
- `Timed out … Developer Mode disabled` → gate #2
- `Device … isn't registered` → missing `-allowProvisioningDeviceRegistration`
- `No profiles … were found … pass -allowProvisioningUpdates` → missing that flag (e.g. used `expo run:ios`)
- `errSecInternalComponent` at Embed Pods Frameworks → gate #3 (user denied keychain; click Always Allow)
- `CoreDeviceError 10002` / `device was not … unlocked` at install/launch → gate #4 (just re-run the launch line once unlocked; the app is already installed)

## 4. Verify what actually shipped

```bash
APP=apps/mobile/ios/build/Build/Products/Release-iphoneos/Huismus.app
codesign -dvvv "$APP" 2>&1 | grep -iE "Authority|TeamIdentifier|Identifier="
security cms -D -i "$APP/embedded.mobileprovision" | plutil -p - \
  | grep -iE "Name|TeamIdentifier|get-task-allow|ProvisionedDevices"
```

Expect `Authority=Apple Development: Jeroen Esseveld`, `TeamIdentifier=5W85L569QN`,
a `get-task-allow=true` **development** profile listing the target device under
`ProvisionedDevices`.

## ⚠️ Distribution caveat — state this when handing over

This is a **development** build: signed for the specific device UDID(s) in the profile, not
uploadable to the App Store or installable on arbitrary phones. Because the team is a **paid
Individual** account (not a free personal team) there's **no 7-day expiry**, but it's still
device-scoped. App Store / TestFlight / ad-hoc distribution is a different signing path
(Apple Distribution cert + a distribution provisioning profile / `xcodebuild archive` +
`-exportArchive`). To watch the running app's logs/behavior, use the `verifier-android`
analogue for iOS or Xcode's device console.
